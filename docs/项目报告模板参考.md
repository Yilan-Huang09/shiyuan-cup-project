# 第一章   需求分析

## 1.1 作品概述

大学生在学业发展、生涯规划与职业准备过程中普遍面临信息分散、路径复杂、决策门槛高等问题。课程安排、绩点管理、竞赛申报、保研考研、实习就业、Offer收集等事项通常分布在学校官网、教务系统、学院通知、招聘平台、社交媒体和学长学姐经验中，信息来源碎片化、时效性不稳定，容易造成遗漏、误判和低效决策[1][[1\]](#_msocom_1) 。本作品旨在打造一个一站式学涯与职涯管理，通过多Agent协作集成教务数据、动态更新竞赛/岗位库、结构化前人经验，解决信息遗漏与路径迷茫问题，实现从“被动接收信息”到“主动规划发展”的转变。

## 1.2 面向人群

本项目面向大学生群体，构建一个具备多智能体协作能力的成长辅助平台。系统以“学业规划、生涯决策、信息获取、任务提醒、持续跟踪”为主线，通过多智能体分工协作，帮助学生完成课程规划、竞赛筛选、保研考研路径分析、就业方向建议、校园通知整合与待办提醒，形成从信息获取到行动执行的闭环服务。

## 1.3 功能需求

系统主要功能包括以下五个方面：

**1.** **学业规****划模块**

自动同步或一键导入当前已修课程及成绩，提供GPA动态测算与学分预警。基于培养方案提供选课建议；支持上传课程大纲，由大模型自动生成阶段性复习提纲与知识点拓扑图。

**2.** **生涯规划模块**

录入目标院校或企业，系统分析历年录取/录用标准，生成当前能力雷达图与目标差距分析。根据日常记录的学业和项目经历，利用LLM辅助生成并润色适用于不同场景的个人陈述（PS）或求职简历。

**3.** **聚合信息引擎**

突破信息孤岛，全天候监控主流科研竞赛官网、高校招生网及知名企业招聘页。过滤无效或过期信息，提取诸如“报名时间”、“参赛要求”、“考核科目”等结构化字段，形成标准化卡片供用户查阅。

**4.** **系统联动与工作台**

适配微信、钉钉、主流日历应用，实现双向日程同步。提供随时记录灵感、待办事项的快捷入口，系统自动将其归类至相应的长期项目下。

**5.** **个性化伴学与动态调整**

系统随用户年级的增长动态切换服务重心（如大二偏重竞赛与基础课，大三偏重复习备考与信息收集）。在经历考试周或大型项目结项后，智能体主动发起对话，引导用户进行经验复盘，更新其个人能力数据库。[[2\]](#_msocom_2) 



图1 功能需求图

## 1.4 非功能需求

在性能方面，系统要求具备较好的响应速度、较低的部署门槛和良好的可扩展性。相较于通用问答型大模型，本作品不止提供回答，更强调对于学生的持续辅助和任务闭环；相较于单纯的信息平台或待办工具，本作品具备意图理解、信息整合与主动提醒能力。

## 1.5 竞品分析

当前大学生学业与生涯规划领域的竞品主要包括传统教务系统、社交媒体平台和招聘平台。整体来看，三者各有侧重但均存在明显不足：

教务系统数据权威但信息局限于校内，缺乏外部资源整合与决策支持；社交媒体信息丰富但高度碎片化，真实性参差不齐且存在营销干扰，难以形成系统性指导；招聘平台在求职阶段匹配效率高，但服务范围局限，无法覆盖完整学业周期。总体而言，现有产品普遍存在信息割裂、时效性不足、决策能力弱以及用户生命周期覆盖不完整等问题，难以满足学生长期、系统化的发展需求。因此，市场亟需一种能够整合多源信息、提供结构化分析与持续路径规划的智能化解决方案。

# 第二章   概要设计

本项目采用“前端交互层—智能体协同层—技能服务层—数据与平台接入层”的总体架构。前端交互层负责用户输入、结果展示、任务提醒和成长档案管理；智能体协同层由多个角色化智能体组成，负责需求理解、任务拆解与协同决策；技能服务层封装课程规划、信息检索、推荐分析、待办生成、日程提醒等能力；数据与平台接入层则连接知识库、校园通知、竞赛信息源、招聘信息源以及第三方校园工作台。

**1.** **整体规划**

系统中的核心设计思想是引入Multi-Agent协同机制[2][[3\]](#_msocom_3) 。不同智能体聚焦不同任务，例如学业规划智能体负责课程与学习建议，生涯规划智能体负责升学与就业分析，信息检索智能体基于RAG和在线搜索，负责抓取和筛选竞赛、政策、Offer与通知信息，提醒管理智能体负责生成待办与日程提醒。主控智能体接收用户需求后，根据任务类型调度子智能体完成协作，并最终汇总为统一输出结果。

在人机交互方面，系统采用本地部署的OpenClaw集成智能体，因此用户可采用对话式交互与结构化展示相结合的方式，既可以通过自然语言直接提出需求，也可以在界面中查看系统整理后的课程规划结果、竞赛推荐列表、升学路径建议、近期待办事项和成长进度记录。

**2.** **技术路线**

技术上，前端采用React构建多端响应式界面[[4\]](#_msocom_4) [[5\]](#_msocom_5) ，后端采用Python实现智能体调度与技能服务，知识增强模块基于向量检索技术实现，外部数据通过 API、校园平台接口或定制化采集模块接入。整体架构模块化程度高，便于后续在不同高校、不同专业场景中快速扩展。

# 第三章   详细设计[[6\]](#_msocom_6) 

## 3.1 界面设计

系统界面以清晰、高效和低学习成本为原则，围绕大学生高频使用场景设计四个核心区域。第一是智能对话区，用户可直接输入诸如“帮我规划下学期课程”“最近有哪些适合我的竞赛”“我该准备保研还是考研”等自然语言需求；第二是信息展示区，用于结构化展示系统分析结果，包括推荐竞赛、课程规划、政策摘要、Offer整理与路径建议；第三是任务提醒区，用于展示系统自动生成或用户手动确认的待办事项、DDL、考试时间与报名节点；第四是个人成长档案区，用于记录用户阶段目标、近期进展、历史建议与成长轨迹。

典型使用流程如下：用户输入需求，主控智能体识别任务类别并分配相应子智能体；相关技能模块完成信息检索、分析推理与结果生成；系统将结果以对话解释与结构化卡片两种形式反馈给用户；若任务中包含关键时间节点，系统可自动生成提醒任务并同步至待办模块。

## 3.2 关键技术

### 3.2.1 基于RAG的个体长期记忆增强机制

在大学生学业与生涯发展场景中，系统面对的并不是一次性的通用问答，而是一个跨学期、跨阶段持续演化的长期辅助过程。用户的目标、经历、状态与需求会不断变化，因此，系统不仅要“回答当前问题”，更要能够结合用户过往信息，持续生成前后一致、贴合个人背景的建议。基于此，本项目设计了基于 RAG 的个体长期记忆增强机制[3][[7\]](#_msocom_7) ，将检索增强生成从通用知识补充扩展为面向个体成长过程的长期记忆支持能力。

**1.** **记忆重构与上下文融合**

该机制以用户的个人成长信息为核心对象，围绕基础画像、阶段目标、重要经历、历史咨询记录、关键时间节点、待办事项以及阶段性成果等内容，构建结构化记忆库与向量化索引。在每次交互中，系统会根据当前任务语义，动态检索与问题最相关的个人记忆片段，并与必要的通用知识共同提供给大模型，从而实现“基于个人历史语境的生成”。相比仅依赖大模型上下文窗口的方式，这种机制能够显著增强系统对用户长期信息的保持能力，使建议在跨轮次、跨阶段使用过程中保持连续性与个性化。

这一设计所解决的核心问题有两个。大模型本身缺乏稳定、持久的长期记忆能力，难以在长期陪伴型场景中持续保留用户的重要背景信息；其次，学业规划、生涯决策、任务提醒等问题具有很强的个体依赖性，如果仅基于当前对话进行推理，容易输出模板化、泛化甚至前后矛盾的建议。通过将 RAG 与 Memory 机制结合，系统把“用户是谁、处于什么阶段、已经做过什么、接下来要做什么”纳入生成过程，提升建议的针对性、一致性与可执行性。

**2.** **信息动态沉淀与召回策略**

本模块的关键技术难点并不在于简单检索，而在于长期记忆的选择、更新与精准召回。如果记忆写入过于宽泛，系统容易引入噪声信息，造成检索污染；如果记忆保留过于有限，则无法支撑真正有效的长期个性化服务。因此，本项目采用“核心信息沉淀 + 阶段状态增量更新”的策略，对记忆内容进行筛选与维护：优先保留对后续规划、决策和提醒有显著影响的信息，如长期目标、关键成果、重要节点、未完成事项以及偏好变化等；对于短期、低价值或重复信息，则不进入长期记忆层。与此同时，在检索阶段，系统会依据当前任务类型与语义上下文，对个人记忆进行相关性匹配与优先级排序，确保召回内容真正服务于当前决策，而不是机械堆叠历史信息。

因此，本项目在 RAG 系统应用的创新点在于：其目标不再是为模型补充外部事实知识，而是构建一种面向大学生成长场景的长期个体记忆增强能力。也就是说，RAG 在系统中承担的不是传统意义上的知识问答辅助模块，而是连接“用户历史—当前需求—未来行动”的关键桥梁。通过这一机制，平台能够在持续使用过程中逐步形成对用户的稳定理解，为后续的任务分解、建议生成、智能提醒与多智能体协同提供统一的个体化信息基础，从而更好地支撑大学生学业与生涯发展的全过程辅助。

### 3.2.2 Skill 技能封装与调用

在本系统中，Skill 不仅仅是功能函数，它是智能体完成具体任务的基础能力单元。我们直接遵循了 Anthropic Agent Skills format 的规范[4][[8\]](#_msocom_8) ，将 AI 的执行逻辑与底层代码解耦，通过“工作手册”的形式显式定义 AI 的行为边界。

为了提升系统的可控性与扩展性，本项目拒绝将能力“写死”在主流程中。我们将课程规划、竞赛检索、政策摘要等能力独立封装为 Skill，并通过统一的配置规则进行注册。这种设计使得 AI 在处理任务时，不仅拥有“大脑”，更拥有一套可随时查阅的“工具使用手册”。每个 Skill 对应一个独立的 .md 描述文件，这与系统根目录下的 AGENTS.md（通用行为准则）共同构成了智能体的运行协议。

在执行流程中，Skill 文档承担了以下关键职能：

l 能力自描述：包含名称、适用场景与能力说明，告知智能体“在什么情况下、为了解决什么问题”而调用。

l 强类型约束：明确定义输入参数与输出格式。这相当于为 AI 制定了写入规范，杜绝了参数幻觉，确保了 Workflow 调用链的数据一致性。

l 执行限制（Constraints）：显式规定调用过程中的禁令与必选项，保障了执行过程的安全稳定。

l 异常回执策略：对失败处理进行了文档化预设，确保智能体在工具失效时能根据手册逻辑进行自我修复或主动追问。

这一设计解决了两个典型问题。第一，若所有任务都依赖单个模型自然语言一次性生成，过程不透明、结果不稳定，也不利于复用；第二，随着功能增多，系统逻辑容易耦合，新增能力时必须频繁修改主流程，维护成本较高。通过将 Skill 抽象为文档化能力模块，系统只需在新增功能时补充对应的 .md 描述和执行接口，即可被智能体识别并纳入调度，无需大幅改动整体框架。

该技术的难点主要体现在描述规范统一性与调用稳定性上。不同 Skill 处理的对象类型不同，有的偏结构化检索，有的偏文本分析，有的偏执行写入。如果描述格式不统一，智能体在做 workflow 编排时容易出现识别混乱、参数不匹配和返回结果不可用等问题。为此，本项目在 Skill 设计中采用统一模板，约束输入字段、输出字段和状态说明，并对异常情况进行显式定义，从而提高调用链稳定性。

本项目在 Skill 设计上的特点，不是简单增加工具，而是将技能作为平台级能力资产进行管理。用 .md 形式组织 Skill，使能力具备较好的可解释性、可扩展性和可维护性，也便于后续面向不同高校和不同业务需求快速增删能力模块。

### 3.2.3 多智能体任务理解、分解与协同决策

大学生在学业发展、生涯规划和阶段决策中提出的需求，通常具有较强的复合性、目标性与连续性，往往难以通过一次简单问答直接解决。例如，“我接下来半年该如何安排才能更有利于保研”这一问题，并不是单一信息检索或单步推理任务，而是同时涉及用户现状识别、目标拆解、约束分析、路径判断、优先级排序以及后续行动安排等多个环节。再如“应该优先准备考研还是先去实习”“下学期应该冲高难课程还是稳住绩点”等问题，本质上也属于多目标约束下的综合决策任务，既需要对问题进行结构化拆解，也需要从多个维度进行权衡判断。若系统仅采用单轮生成方式直接作答，通常只能给出较为宽泛的经验性建议，难以形成清晰、可执行且具有解释性的支持方案。

**1.** **动态工作流构建**

(1) 生成式任务

针对这一问题，本项目设计了面向复杂需求的多智能体任务理解、分解与协同决策机制，并结合 Agent-Skill 架构构建动态工作流。其核心思想参考了 MAS-GPT 的设计理念[5][[9\]](#_msocom_9) ：不是为所有问题预先固定一套静态的多智能体处理流程，而是将“如何针对当前用户需求组织合适的执行结构”本身视为一个生成式任务，由系统根据输入内容动态生成与当前问题相匹配的处理路径。MAS-GPT 强调将多智能体系统的构建统一表示为可执行结构，使 agent 的角色定义、调用关系和执行顺序能够被显式表达。

(2) 生成Plan与Workflow

系统在接收到用户请求后，并不直接进入回答阶段，而是先对问题进行任务级理解，抽取其中的目标、约束条件、时间范围、关键对象与隐含需求，并据此生成面向执行的 plan 与 workflow。其中，plan 用于描述任务总体目标、阶段步骤和执行顺序，回答“这件事应该如何拆解、先做什么、后做什么”；workflow 则进一步将这些步骤映射为具体的执行链路，明确每一步应由哪个 Agent 负责调度、调用哪个 Skill 完成、依赖哪些记忆或数据模块，以及结果如何向下游传递。

**2.** **执行机制**

系统中可设置学业分析智能体、生涯规划智能体、信息检索智能体和任务管理智能体等角色，分别从课程与绩点影响、升学与就业路径、外部事实依据、任务执行成本与时间安排等角度，对同一问题进行分析。首先，由主控智能体根据用户请求生成整体 plan，并识别问题涉及的核心决策维度；其次，将不同分析维度分派给相应角色智能体；随后，各角色智能体基于共享的用户状态、历史记忆、RAG 召回内容及自身职责输出子结论；最后，由主控智能体对这些子结论进行整合，比较其中的一致性、冲突点与支撑依据，并结合用户目标优先级、现实约束和时间条件，形成最终建议，输出“推荐方案 + 理由说明 + 可选替代路径”的结构化结果。

该机制主要解决了三个方面的问题。第一，它降低了复杂需求直接求解时对单次生成能力的依赖，使系统能够从“整体性、模糊化回答”转向“分步骤、分模块处理”，从而提升复杂场景下的稳定性与可控性。第二，它增强了系统对多目标决策问题的处理能力，通过多角色协同分析缓解单一模型容易出现的单视角偏置，使输出结果更加全面、均衡。第三，它提高了结果的结构化程度与可解释性，使系统给出的不只是原则性建议，而是能够进一步落实到阶段任务、优先级顺序、关键节点安排及备选路径上的行动方案。尤其在保研规划、考证安排、竞赛准备、实习申请等真实成长场景中，这种“先拆解、再协同、后裁决”的 workflow 化处理方式更符合实际决策逻辑，也更便于后续持续跟踪与动态调整。

**3.** **核心优势与创新价值**

本项目在这一部分的关键创新，不在于形式化地叠加多个模型或智能体，而在于将复杂用户需求统一转化为“可理解、可拆解、可协同、可裁决”的执行过程。它继承了 MAS-GPT 将系统构建本身视为生成任务的思想，同时进一步将其落地到大学生学业与生涯辅助场景中，形成了面向真实服务过程的 Agent-Skill 协同机制。由此，系统不仅具备“理解问题”的能力，也具备“组织执行与辅助决策”的能力，为后续的 Skill 调用、提醒触发、长期跟踪和多智能体协作提供了统一支撑。

### 3.2.4 基于Openclaw的平台联动与执行机制

大学生在成长过程中面临的大量关键事项具有明显的时间敏感性，例如竞赛报名、课程退补选、考试安排、夏令营申请、材料提交和实习投递等。许多情况下，问题并不在于“学生不知道该做什么”，而是在获得了一些碎片化信息之后，仍停留在了解层面，没有及时转化为行动，最终错过重要节点。因此，系统如果只停留在对话层面，其实际使用价值仍然有限，必须进一步具备从“理解需求、生成方案”走向“触发执行、持续跟进”的能力。

**1.** **核心机制**

针对这一问题，本项目设计了基于 OpenClaw 的平台联动与自动化执行机制。OpenClaw 的核心启发并不只是提供对话能力，而在于它将智能体能力进一步扩展到真实环境中的任务执行与多平台联动，支持通过工具、技能和外部接口完成浏览器操作、消息交互、任务触发与持续运行，使智能体从“回答问题”走向“执行任务”。本项目结合大学生学业与生涯辅助场景，在完成任务理解、方案生成和多智能体决策后，不将结果停留在文本输出层，而是进一步从结果中结构化提取关键时间节点、重要事项和待执行任务，并将其映射为可执行动作链。系统既可以在站内生成待办，也可以在具备接口条件的情况下接入钉钉等外部平台，将任务同步到更贴近真实校园场景的使用入口中，从而形成“自然语言触发—任务抽取—平台写入—结果反馈”的闭环。这里，前序生成的 plan 与 workflow 不仅服务于问题分析，也进一步服务于任务落地，使系统具备持续协助用户推进事项的能力。

这一机制主要解决三个方面的问题。第一，它降低了“知道但忘记执行”的风险，将抽象建议进一步转化为明确的执行任务与时间节点，提高系统输出的落地性。第二，它使平台具备持续服务能力，而不是在一轮咨询结束后立即中断，从而更符合大学生成长任务具有长期性、阶段性和连续性的特点。第三，它增强了系统与外部环境之间的联动能力，使智能体不再局限于封闭式问答，而能够借助平台模块和外部接口参与真实任务推进，这一点也与 OpenClaw 所体现的“工具化、技能化、可执行化”智能体方向是一致的。

**2.** **执行边界与安全风控**

这种基于OpenClaw的联动与自动化执行机制也面临明显的技术挑战。其一，任务抽取必须足够准确，否则系统容易生成大量无关待办，反而增加用户负担；其二，提醒节奏需要合理控制，过于频繁会打扰用户，过于稀疏又难以发挥作用；其三，不同外部平台的接口能力和接入方式存在较大差异，若主流程与平台细节强耦合，会显著增加系统复杂度；其四，一旦系统进入“执行动作”，就必须同步考虑权限边界、执行确认和过程可追踪性等问题。公开资料也显示，OpenClaw这类具备多工具调用和环境执行能力的智能体，在能力增强的同时，对安全控制、权限隔离和审计机制提出了更高要求。为此，系统优先抽取与用户目标直接相关的重要事项，再由用户确认后写入提醒列表或执行队列；外部平台通过独立适配层接入，减少平台差异对主流程的影响；对于关键动作，则保留确认和追踪机制，避免自动化执行越界。因此系统能够在增强任务落地能力的同时，保持执行边界清晰、联动过程稳定。

总体而言，本项目在该部分的特点，是应用 OpenClaw 的“从会回答到会执行”的智能体设计理念，将平台联动与提醒机制视为智能体服务链的重要组成部分，而不是附属功能。通过这一机制，系统能够真正从“给建议”进一步走向“帮执行”，把分析结果转化为可持续推进的任务行动，从而提升平台在大学生成长场景中的现实落地价值与长期服务能力。

### 3.2.5 隐私安全保护

本项目面向大学生学业发展与生涯规划场景，系统在长期使用过程中会处理用户画像、成长目标、咨询记录、待办事项和阶段状态等信息，具有较强的个人敏感性。同时，平台采用本地部署的 OpenClaw 集成智能体，并结合长期记忆、多智能体协作和任务提醒机制，因此隐私安全问题不仅体现在数据是否存储，更体现在数据如何保存、谁可以访问以及执行过程是否受控等方面。

针对上述特点，系统在隐私安全方面主要从数据最小化、分层存储、权限控制和结果可控四个方面进行设计。首先，在数据采集阶段坚持最小必要原则，仅保留支撑个性化服务、长期规划和提醒生成所需的核心信息，避免将无关对话内容和低价值过程信息直接写入长期记忆。其次，在数据管理层，对用户画像、长期目标、咨询记录、待办事项和运行日志进行分类管理，其中高价值且与后续决策密切相关的信息进入长期记忆层，短期上下文和临时结果则保留在会话级存储中，以降低敏感信息集中存储的风险。

在长期记忆机制中，系统采用“写入筛选 + 检索过滤”的控制方式：只有对后续规划具有持续价值的信息才进入长期记忆；在后续调用时，也仅检索与当前任务相关的记忆内容，避免无关历史信息被重复暴露。在多智能体协作过程中，不同模块按照职责范围访问数据，例如学业规划模块主要处理课程与学习目标，提醒模块主要处理时间节点与待办信息，减少多模块之间对完整用户信息的无差别共享。必要时可结合基于角色的权限控制机制，对不同功能模块的读写范围进行限制。

此外，针对技能调用和任务执行过程，系统采用受限执行思路，通过 sandbox 沙箱机制限制技能运行时的目录访问、接口调用和执行范围，降低任务执行过程中的越权风险。在前端展示与结果输出阶段，系统尽量采用摘要化、结构化和必要脱敏的方式呈现结果，避免在多轮交互中反复回显完整敏感信息；对于待办写入和提醒生成，则优先采用“系统建议—用户确认—再写入”的方式，增强用户对个人信息和任务数据的控制能力。

本项目的隐私安全设计与本地部署、多智能体协作、长期记忆增强和任务提醒机制协同考虑，通过最小必要采集、分类存储、按需访问、sandbox 受限执行和用户可控写入等方式，在保证持续个性化服务能力的同时，提高系统在大学生学业与生涯发展场景中的安全性与可信度。[[10\]](#_msocom_10) [[11\]](#_msocom_11)

针对系统可能面临的校园高峰期（如期末复习周、全校选课季），测试模拟了不同梯度的并发用户请求。结果显示：在并发数低于20时，系统的平均TTFT基本保持在1.5s以内，Token生成速率未出现明显衰减；当并发数攀升至50极限值时，受限于外部API速率与算力瓶颈，TTFT增至约3.8s，吞吐量缓慢衰减至约18Tokens/s。极端高压下，系统进程调度稳定，全程未触发请求熔断或内存溢出现象，证明当前架构具备良好的降级容错能力。

**3.** **上下文窗口长度与延迟相关性分析**

鉴于“长期记忆增强机制”会随用户在校时长的增加而持续累积数据，测试专门分析了长文本输入对响应速度的拖累效应。实测表明，当上下文携带的Token数量从1k激增至8k时，TTFT仅呈现微弱的线性增长趋势（每增加1k Tokens，耗时仅增加约0.15s）。

此数据验证了系统中“核心信息沉淀”策略的有效性，通过在记忆写入前置清洗高频冗余语料，精准缩减了检索引擎的排查范围，确保高年级用户在调取庞大个人档案时依然能够获得高质量的响应体验。

**1.** **提示词注入测试**

测试结果：得益于系统在OpenClaw框架前置部署的意图过滤器，结合系统提示词的强制闭环边界锁定，系统对100条不同变种的恶意注入请求实现了100%的拦截率。智能体能够准确识别越权意图，礼貌拒绝违规请求，并主动引导对话回归学业规划的正轨，充分保障了生成内容的纯洁性与学术规范性。

**2.** **智能体工具调用越权与沙箱测试**

测试结果：系统的RAG检索模块在底层不仅执行了标准的RBAC机制，更在向量检索层实现了基于User_ID的租户级命名空间硬隔离。任何跨域检索的恶意请求，在网关的元数据过滤阶段即被直接阻断，根本不会被召回进入大模型的上下文窗口，从物理思路上杜绝了隐私泄露的可能；面对试图骗取系统底层配置的复杂Prompt注入，依托于OpenClaw框架的输出校验机制，系统成功拦截了所有违规探测。

测试结果表明，系统在面临针对RAG记忆库的恶意探测时，能够100%保证当前会话的“数据气密性”。大模型无法获取、拼接或泄露任何第三方敏感数据与系统底层结构，数据安全隔离性完全符合大语言模型应用的安全合规标准。

 



 

# 第五章   安装及使用

本项目前端基于Node.js 18+、Vite 6与React 19构建，后端基于Python 3.11 +与FastAPI、OpenClaw引擎与Redis开发，支持跨平台部署，以Ubuntu 22.04为例说明；因涉及大模型API与隐私数据，建议启用HTTPS。后端建议配置不低于4C8G，需稳定外网以对接阿里云通义千问qwen3.5-flash服务；前端静态资源可托管于Nginx，部署时需将访问域名加入后端CORS允许列表。本说明文档为求精简已忽略部分常规系统优化步骤，请根据实际部署环境自行调整。

## 5.1 后端部署

### 5.1.1 环境装配流程

将 ZhiTu_Backend.zip 上传至服务器并解压，进入后端项目根目录后创建独立虚拟环境，并安装项目依赖。流程如下：

cd ~/

unzip ZhiTu_Backend.zip

cd ZhiTu_Backend

python -m venv .venv

pip install --upgrade pip

pip install -r requirements.txt

部署人员需基于模板创建生产环境配置文件，并填写必要参数：

sudo apt update && sudo apt install redis-server -y

sudo systemctl enable --now redis-server

cp .env.example .env && vim .env

APP_NAME=ZhiTu_Intelligence_Platform

APP_ENV=production

APP_HOST=0.0.0.0

APP_PORT=8000

API_V1_PREFIX=/api/v1

CORS_ORIGINS=http://your_domain.com,http://localhost:3000

 

LLM_API_KEY=sk-xxxxxxxxxxxxxxxxxxxx

### 5.1.3 服务启动与可用性验证

完成配置后，可通过 uvicorn main:app --host 0.0.0.0 --port 8000 启动后端服务。服务启动后，可访问 http://服务器IP:8000/health 检查系统是否正常运行，并访问 http://服务器IP:8000/docs 查看自动生成的接口文档，验证工作台、学业规划、日程管理及多智能体交互等模块是否完成加载。若启用完整 AI 模式，还可结合日志输出进一步确认模型服务连接、智能体执行链及 Skill 技能配置是否生效。

## 5.2 前端部署

### 5.2.1 环境装配流程

将前端源码包 ZhiTu_Frontend.zip 上传至服务器并解压。建议在独立的工作目录下执行构建流程：

cd ~/

unzip ZhiTu_Frontend.zip

cd ZhiTu_Frontend

npm install

npm run build

执行完成后，系统将在根目录下生成 dist 目录，该目录即为可用于生产环境部署的静态资源。

### 5.2.2 生产环境参数配置

部署人员需配置站点访问地址，以确保前端能够正确识别资源路径：

cp .env.example .env

vim .env

在 .env 文件中配置如下核心参数：

VITE_APP_URL=https://zhitu.yourdomain.com

VITE_API_BASE_URL=https://api.zhitu.yourdomain.com/api/v1

### 5.2.3 生产级发布与可用性验证

构建生成的dist目录可部署至Nginx、Vercel或静态托管平台。若在服务器本地进行预发布测试，可执行：

npm run preview -- --host 0.0.0.0 --port 4173

服务发布后，应通过以下检查清单验证部署完整性：

访问首页，确认TailwindCSS4样式与TypeScript编译后的JS脚本无404错误；检查工作台、学业规划及日程管理模块是否正确获取后端种子数据；打开Agent抽屉界面，验证“小知”虚拟助手能否正常唤起，并测试多智能体协同流程的UI响应是否顺畅；确认Skill技能手册展示区域能够正确渲染由后端动态下发的.md描述文件。

## 6.2 项目成果与创新点

本项目最终产出了一套轻量、高效、可扩展的“知途”多智能体辅助平台。我们摒弃传统的静态逻辑脚本，引入MAS-GPT设计理念，使系统能够根据用户的动态自然语言指令，实时组织Specific MAS执行流，实现了架构层面的高度灵活性与任务适应性。在此基础上，项目的核心创新点体现在以下方面：

针对大学生规划需求复合性强、场景多变的特点，系统摒弃了传统的静态工作流，引入了基于自然语言驱动的动态多智能体协作模型。主控智能体能够实时解析用户意图，动态拆解任务。此架构设计显著降低了系统各功能模块间的耦合度，提升了复杂决策任务的处理效率与系统的横向扩展能力。

为解决大模型在长周期、多轮交互中容易丢失上下文的工程痛点，团队将检索增强生成RAG技术拓展为个体成长档案。通过将用户的专业背景、阶段目标、历史决策及待办事项进行向量化存储与索引，系统在每次交互前动态召回相关个人记忆。该机制使得AI能够突破上下文窗口限制，在跨学期、跨阶段的使用中保持认知连贯性，提供符合用户历史语境的连续性指导。

系统利用多智能体的异构特性，将复杂的规划问题拆解分发。不同智能体分别从“校内绩点约束”、“校外考研/竞赛态势”等单一维度进行独立评估，最终由主控节点进行冲突检测与加权整合。这种分布式推理机制有效缓解了单一LLM在处理复杂逻辑时容易产生的幻觉与视角偏置，显著提升了输出方案的客观性与专业度。

为克服传统AI助手“重建议、轻落地”的局限，本项目将核心执行能力封装为标准化的Skill工具如时间流计算、外部接口请求等赋予智能体调用外部系统的权限。通过与校园工作台（如钉钉Webhook）的底层接口联动，系统能够自动从规划方案中提取高优先级的截止日期和复习节点，将其转化为强类型任务直接推送到用户终端，完成完整工程闭环。

## 6.3 项目价值

在教育普惠与社会公平层面，本项目的核心价值在于通过技术手段消除长期存在的信息不对称现象。传统学业指导高度依赖零散的经验数据或高成本的人工咨询，而知途平台通过全天候运行的智能辅导系统，为不同资源背景的学生提供均等化的学业与职业规划指导。

该系统能够辅助学生在冗杂的数据中高效定位目标，并依赖多智能体协同机制输出基于客观逻辑的规划建议，引导学生由无序决策转向系统性规划。这从机制上降低了大学生在升学与就业过程中的决策不确定性，并为高等教育的个性化辅导体系提供了创新的技术支撑。[[25\]](#_msocom_25) 

在技术工程维度，本项目验证了异构多智能体架构在处理复杂约束决策时的计算优势。通过将大语言模型的生成能力与标准化外部技能库融合，系统成功将非确定的模型推理转化为确定的业务执行。该架构不仅实质性提升了数字化教育工具的交互效能，也为原生人工智能校园应用的开发构建了可复用的参考范式与实证数据。

在系统落地与商业转化方面，项目展现出极高的经济效能与适配性。采用轻量化容器部署与云端应用接口架构，使应用主体无需配置高规格硬件即可维持系统运转，实现了算力成本的最优化。同时，智能体模型与业务逻辑的深度解耦设计，使得平台能针对不同高校需求进行快速定制与知识库迁移，具备显著的软件即服务（SaaS）扩展潜力。通过多智能体协同技术[6][[26\]](#_msocom_26) ，系统有效将政策要求的“人工智能+教育”从宏观指导转化为可操作的学业辅助工具，实质性地推动了校园数字化向智能化、个性化阶段的跨越。

## 6.4 经验总结与未来展望

团队通过本项目深刻体会到 AI 原生应用的工程挑战：核心不再仅是编写确定性业务逻辑，而是约束大语言模型的非确定性。实践证明，清晰的角色边界、标准化的技能接口与稳健的前置提示词机制是保障系统鲁棒性的关键。此外，从单轮问答向多智能体协同架构的演进表明，实用的校园辅助工具必须具备任务拆解与平台执行能力，而非仅停留在文本交互[[27\]](#_msocom_27) [7]。
