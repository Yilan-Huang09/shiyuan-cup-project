# Aurora Classroom 项目计划书 / 项目报告六章设计文档

## 目标

为 `Aurora Classroom · 智慧课堂交互演示系统` 生成一份可直接扩写为竞赛项目计划书 / 项目报告的六章写作蓝图。文档目标不是直接代替终稿，而是统一全书定位、事实边界、章节结构、写作重点与审查标准，作为后续成文的唯一设计依据。

## 范围

- 产出六章连续写作 spec
- 将全书拆分为六个章节任务，逐章可独立写作与审查
- 统一“当前仓库事实”与“完整系统设计口径”的边界
- 为图表、章节逻辑与竞赛表达提供稳定骨架

## 任务拆分

- Task 1：第一章《需求分析》
- Task 2：第二章《概要设计》
- Task 3：第三章《详细设计》
- Task 4：第四章《测试报告》
- Task 5：第五章《安装及使用》
- Task 6：第六章《项目总结》

## 信息来源

本设计文档以以下材料为唯一事实基础：

- `docs/项目报告模板参考.md`：只借章节结构、竞赛语体与图表组织方式

## 全局写作约束

### 1. 项目定位

- 全书统一定位为：`面向智慧课堂场景的交互演示系统 / 前端原型`
- 不得写成“已全面落地的智慧课堂平台”
- 允许写“完整系统设计口径”，但必须明确标注为未来扩展或后续部署方向

### 2. 事实边界

- 当前仓库强事实：前端交互演示系统、课堂脚本化流程、多角色对话呈现、知识白板、来源抽屉、碰撞 / 冲量模拟器、随堂测验、角色配置入口
- 当前仓库弱事实或未证实部分：真实 RAG 检索、真实多智能体编排、教师资料自动入库、课堂数据闭环、账号体系、后端部署、商用落地
- 所有未证实内容只能写成“设计思路”“完整系统部署口径”“未来展望”

### 3. 名称与术语统一

- 系统全名统一为：`Aurora Classroom · 智慧课堂交互演示系统`
- 正文默认使用职责化表述：`教师 Agent`、`学生 Agent`、`助教 Agent`、`总结 / 资料整理 Agent`
- 若引用代码事实，可说明当前实现角色为 `teacher / student_a / student_b / assistant`
- `RAG` 在正文中默认解释为“检索增强生成（Retrieval-Augmented Generation）”的设计方向；若讨论当前仓库事实，应收敛为“来源引用与知识边界可视化形态”

### 4. 文风约束

- 参考模板的竞赛语气、问题-方案-价值链、图表组织方式
- 不继承模板中的“知途 / 大学生成长规划 / 保研就业 / 校园工作台”业务叙事
- 不得借用模板中的量化数据、问卷结果、性能指标、部署规模与上线结论

### 5. 全书红线

- 不得把 `mockData.ts` 预设流程写成真实在线多智能体协同结果
- 不得把 `SourceDrawer` 的 UI 交互写成真实课程知识库与向量检索已完成
- 不得捏造学校试点、教师反馈、学生口碑、学习成效、SUS 分数、商用部署
- 不得把前端交互演示系统写成完整全栈系统

## Task 1 第一章《需求分析》

### 章节定位

第一章负责回答“为什么做、给谁做、要解决什么问题、系统应提供哪些能力、与现有方案差异在哪里”。本章重点不是技术实现，而是把 `Aurora Classroom` 从课堂痛点出发定义为一个面向智慧课堂的交互演示系统原型。

### 建议结构

- 1.1 作品概述
- 1.2 面向人群
- 1.3 功能需求
- 1.4 非功能需求
- 1.5 竞品分析

### 核心写作要点

- 1.1 从传统课堂互动不足、通用 AI 幻觉与缺乏课堂组织能力切入，将项目定义为“多角色课堂研讨 + 知识来源锚定 + 交互实验 + 白板沉淀”的智慧课堂交互原型
- 1.2 主用户锁定教师与学生；教师强调资料边界控制、讨论组织、知识展示与过程引导；学生强调多视角讨论、概念可视化理解、随堂练习与即时反馈
- 1.3 功能需求统一抽象为六类：多智能体课堂对话、知识卡片白板、知识来源抽屉、交互式物理模拟器、随堂测验、角色配置与课堂流程编排
- 1.4 非功能需求聚焦交互实时性、展示清晰度、易用性、可解释性、可扩展性、兼容性、演示稳定性、教学可控性
- 1.5 竞品对标通用 AI 问答工具、传统智慧课堂/电子白板平台、单点仿真/题库工具，核心差异化为“以教师材料为知识边界的多角色课堂研讨”

- 课堂活动闭环图
- 用户与痛点表
- 功能需求总表
- 非功能需求表
- 竞品对比表

### 本章边界

- 不写成已落地全栈平台
- 不把 `mockData` 流程写成真实自治 Agent
- 不把来源抽屉写成真实后端 RAG 已落地

### 通过标准

- 1.1-1.5 逻辑闭环完整
- 主线聚焦智慧课堂
- 功能与价值表达与仓库事实一致
- 完整版能力均以设计口径表达

## Task 2 第二章《概要设计》

### 章节定位

第二章负责把第一章需求转译为“系统如何实现”的总体方案。章节目标是搭起从课堂问题到系统结构的桥梁，让评审看清系统目标、架构层次、模块关系与课堂业务闭环。

### 建议结构

- 2.1 系统整体规划
- 2.2 技术路线与选型
- 2.3 总体架构与核心模块关系
- 2.4 核心业务闭环与运行机制

### 核心写作要点

- 2.1 将系统定义为“面向智慧课堂场景的交互演示系统原型”，目标是围绕教师资料边界、课堂互动组织、知识可视化与即时反馈构建课堂底座
- 2.2 前端事实层写 `React 19 / TypeScript / Vite 6 / Tailwind CSS 4 / Motion / KaTeX` 与“三域协同界面”；智能能力层只写设计口径
- 2.3 建议四层架构：表现交互层、课堂编排层、智能服务层、知识数据层
- 2.4 业务闭环统一为“教师供给资料 -> 系统组织课堂主题 -> 多角色研讨 -> 白板沉淀知识 -> 实验验证 -> 测验反馈 -> 课后沉淀”
- 强调系统不是普通聊天窗口，而是“对话驱动白板、引用、实验与测验同步更新”的复合课堂系统

## Task 3 第三章《详细设计》

### 章节定位

第三章负责把第二章的总体方案落到可感知的界面与可论证的技术机制上。其作用是说明“课堂到底如何被看见、被操作、被组织”，并为第四章测试提供可验证对象。

### 建议结构

- 3.1 界面设计
- 3.2.1 教学资料锚定与来源溯源机制
- 3.2.2 多角色课堂协同机制
- 3.2.3 知识白板与卡片编排机制
- 3.2.4 交互仿真与测验一体化机制
- 3.2.5 角色配置、扩展能力与安全边界

### 核心写作要点

- 3.1 将界面定义为“课堂交互工作台”，采用“左侧资料抽屉 - 中部知识白板 - 右侧 Agent 对话区”三域协同
- 3.1 突出动态知识白板是最强差异化亮点：不是普通聊天界面，而是对话驱动白板生成、资料联动溯源、实验嵌入和测验反馈的课堂式界面
- 3.2.1 当前事实层写知识抽屉、引用按钮、来源预览和来源字段挂载；完整 RAG 仅作为未来机制上升表达
- 3.2.2 当前事实层写 Teacher / Student A / Student B / Assistant 的角色化对话、消息队列、打字动画与配置浮层；真实 Agent Team 协作属于设计口径
- 3.2.3 重点写卡片类型化渲染、拖拽吸附、缩放、固定、折叠、编辑、避碰与空位搜索，并把这些机制解释为“课堂板书自动整理机制”
- 3.2.4 写冲量模拟器、碰撞模拟器、实时公式计算、5 题测验、即时判分与解析，构成“讲解 - 演示 - 练习 - 回看”的一体化课堂机制
- 3.2.5 当前只写前端配置入口和演示系统的零持久化边界；Skill 化编排、平台联动、数据治理、权限隔离属于后续扩展

### 图表建议

- 总界面分区图
- 课堂 8 步推进图
- 来源溯源与抽屉联动图
- 白板卡片编排与避碰机制图
- 仿真 / 测验一体化流程图
- 关键技术支撑表

### 本章边界

- 不把 `mockData.ts` 预设课堂脚本写成实时生成
- 不把 `SourceDrawer` 的上传 / 导入 UI 写成真实知识库建库
- 不把角色配置 UI 写成真实 Prompt 编排后台

### 通过标准

- 评审能清楚理解界面结构、课堂流程和关键技术机制
- 最强事实点聚焦白板系统、仿真系统和多角色界面

- 4.2 交互性能与运行稳定性测试

## Task 6 第六章《项目总结》

### 章节定位

第六章负责收束全书：总结项目实施过程、阶段性成果、项目价值与后续方向。章节目标不是宣布“项目已全面完成”，而是对前端智慧课堂交互原型的阶段性成果进行可信总结。

### 建议结构

- 6.1 项目实施过程
- 6.2 项目成果与创新点
- 6.3 项目价值
- 6.4 经验总结与未来展望

### 核心写作要点

- 6.1 从课堂教学痛点切入，说明团队主动选择高中物理“动量定理 / 动量守恒”作为首个演示场景，以这一知识点兼具公式推导、概念辨析、动态过程展示和随堂测验组织的综合优势，系统化验证智慧课堂交互框架在真实教学问题中的表达能力
- 6.1 写需求收敛、场景建模、界面框架、课堂脚本编排、模块集成与演示验证的实施路径
- 6.2 总述成果形态：项目已形成可运行、可构建、可本地演示的智慧课堂交互前端原型
- 6.2 创新点聚焦多角色课堂交互、动态知识白板、交互式物理模拟、测验与引用同屏闭环，不夸大为算法或平台突破
- 6.3 价值部分写教学价值、学习价值、教师价值、工程价值、展示与竞赛价值，结论统一降格为“阶段性验证价值”
- 6.4 经验总结强调：课堂流程组织比单纯接模型更重要；界面与交互编排本身就是教学设计；前端原型先行是一种有效验证路径
- 6.4 未来展望单独降格写真实 RAG、多 Agent、多学科扩展、课堂数据闭环、教师后台与平台接入
