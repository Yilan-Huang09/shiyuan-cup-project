# Aurora Classroom · 智慧课堂交互演示系统项目报告

# 第一章   需求分析

## 1.1 作品概述

在当前课堂教学场景中，教师普遍面临两个长期存在的问题。其一，课堂讲解、板书整理、实验演示、资料引用与随堂检测往往分散在多个工具之中，教学节奏容易被界面切换打断。其二，通用大模型虽然能够生成自然语言回答，但缺乏明确的课堂知识边界与教学组织能力，容易出现内容泛化、来源不清和课堂节奏失控等问题，难以直接承担严肃教学场景中的课堂协作角色。

针对上述问题，`Aurora Classroom · 智慧课堂交互演示系统` 将系统目标定义为一个面向智慧课堂场景的交互演示系统原型。系统围绕“教师资料边界控制、多角色课堂研讨、知识白板沉淀、来源锚定展示、交互式实验验证、随堂测验反馈”六条主线展开，尝试将课堂中的讲授、提问、推演、演示和练习组织到同一交互工作台中，使教学活动不再停留于单一聊天窗口，而是转化为可视化、可回看、可操作的课堂过程。

当前仓库对应的是一个前端高保真演示原型，而不是已经完成后端部署的全栈智慧课堂平台。系统通过预设课堂脚本、消息队列、白板卡片、来源抽屉和物理仿真组件，验证“对话驱动课堂演示”的核心交互链路是否成立。该实现路径强调先验证课堂组织方式，再逐步扩展到真实 RAG、真实多 Agent 编排和后端数据闭环。

在教学主题选择上，团队主动选择高中物理“动量定理 / 动量守恒”作为首个演示场景。该知识点同时具备公式推导、概念辨析、动态过程可视化和随堂测验组织等特点，适合作为智慧课堂交互框架的首个综合验证样本。当前版本的 8 步课堂流程即围绕这一主题展开，用于验证系统在真实教学叙事中的表达能力与交互连贯性。

## 1.2 面向人群

本项目主要面向中学课堂教学场景中的教师与学生两类核心用户。

对于教师而言，系统的核心价值不在于替代教学，而在于帮助教师将课堂内容组织得更清晰、展示得更集中、引用得更可信。教师希望系统能够围绕自己选定的教学资料展开讲解，在课堂推进过程中同步生成知识要点、呈现关键公式、突出易错点，并在需要时嵌入实验演示与随堂练习，以降低课堂切换成本并提高讲解的可控性。

对于学生而言，系统的价值在于把抽象知识点转化为多视角、可操作、可回看的课堂体验。学生既能通过多角色对话看到不同提问路径，也能通过白板卡片、碰撞演示与测验反馈完成概念理解、规律辨析与即时巩固，从而提升对知识结构和问题求解过程的理解深度。

表1 面向人群与核心诉求表

| 用户角色 | 典型痛点 | 核心诉求 |
| --- | --- | --- |
| 教师 | 课堂材料分散、板书与演示割裂、来源引用不直观 | 统一组织课堂内容，增强展示性与课堂可控性 |
| 学生 | 抽象概念难理解、课堂互动被动、知识点难沉淀 | 获得多视角讨论、动态演示与即时反馈 |

## 1.3 功能需求

围绕智慧课堂场景，本项目将系统需求抽象为以下六类核心功能。

**1. 多角色课堂对话功能**

系统需要具备多角色课堂对话展示能力。当前原型通过教师、学生 A、学生 B、助教等角色的消息队列，模拟课堂中的提问、补充、辨析与总结过程，使课堂表达不再是单向讲授，而是形成具有节奏感的讨论链路。

**2. 知识卡片白板功能**

系统需要具备课堂知识可视化沉淀能力。对话过程中出现的关键概念、核心公式、易错点、小结内容和互动组件，需要被同步转化为白板卡片，并支持拖拽、缩放、固定、折叠和编辑，以模拟课堂板书与知识整理过程。

**3. 来源抽屉与引用溯源功能**

系统需要支持资料边界展示。教师讲解或系统提示中涉及的重点内容，应能通过引用按钮关联至材料来源，并在抽屉中展示对应预览内容，使学生能够直观看到“这条结论来自哪里”。

**4. 交互式物理仿真功能**

系统需要支持课堂中的实验性演示。当前实现重点覆盖完全非弹性碰撞与完全弹性碰撞两类交互模拟，并在组件层实现了冲量-动量变化模拟器，用于承载“公式推导 - 参数调整 - 过程观察 - 结果验证”的课堂活动。

**5. 随堂测验与反馈功能**

系统需要支持讲练结合的课堂闭环。当前原型内置 5 道围绕动量定理与动量守恒的选择题，支持逐题作答、即时判分和解析展示，用于完成课堂尾声的快速巩固与反馈。

**6. 角色配置与课堂流程编排功能**

系统需要支持课堂角色与展示逻辑的基础配置。当前版本提供角色名称、人设指令占位和知识库引用偏好等配置入口，并通过底部候选按钮驱动课堂步进，以验证课堂编排机制是否清晰可用。

## 1.4 非功能需求

除核心功能外，系统还需要满足课堂演示场景下的非功能需求。首先，系统应具备良好的交互实时性，课堂步进、消息切换、卡片生成和抽屉展开不能造成明显卡顿。其次，系统应强调展示清晰度与教学可读性，避免信息堆叠造成课堂认知负担。再次，系统应具备较高的可解释性和教学可控性，使教师能够清楚地知道当前内容来自何处、系统正在推进什么、哪些内容属于演示、哪些属于课堂资料。

与此同时，系统还需要保留一定的可扩展性和兼容性。当前版本虽以前端原型为主，但整体结构应支持后续接入真实知识库、智能体调度服务和教师后台。前端运行环境应尽量轻量，确保在常规浏览器和常见开发环境中即可完成演示与验证。

## 1.5 竞品分析

当前与本项目相关的产品大体可以分为三类。第一类是通用 AI 问答工具，其优势在于自然语言理解与快速回答，但其课堂知识边界和教学组织能力较弱。第二类是传统智慧课堂或电子白板平台，这类产品在课堂展示、课件投屏和板书管理方面较成熟，但通常缺乏多角色智能互动与知识来源联动能力。第三类是单点式仿真平台或题库系统，这类工具能够解决实验演示或练习巩固中的某一环节问题，但很难独立构成完整课堂流程。

`Aurora Classroom` 的差异化不在于单一能力点的堆叠，而在于把多角色讨论、知识沉淀、来源锚定、实验演示与测验反馈整合为一个课堂工作台。它不是传统聊天机器人，也不是单纯的电子白板，而是一个围绕课堂推进逻辑组织的交互式教学原型。

| --- | --- | --- | --- | --- |
| 课堂组织能力 | 强，围绕课堂步进组织 | 弱，以单轮问答为主 | 中，偏展示与投屏 | 弱，聚焦单功能 |

| 板书沉淀能力 | 有动态卡片白板 | 弱 | 有，但智能联动较弱 | 弱 |
| 演示与练习闭环 | 有 | 弱 | 中 | 通常只覆盖单环节 |

# 第二章   概要设计

本项目采用“表现交互层 - 课堂编排层 - 智能服务层 - 知识数据层”的总体架构。当前仓库已实现的重点集中在前端表现交互层与部分课堂编排逻辑，用于完成智慧课堂交互流程的可视化验证；智能服务层与知识数据层的完整能力，则在本章中作为可扩展设计口径进行说明，而不将其误写为当前仓库既成事实。

## 2.1 系统整体规划

`Aurora Classroom` 的整体目标，是构建一个面向智慧课堂场景的交互演示系统原型。系统并不把课堂理解为“用户向模型发问，模型给出回答”的单点过程，而是把课堂视为由资料、角色、板书、实验、练习和反馈共同构成的复合型交互过程。因此，系统从一开始就采用三域同屏的工作台形态，将知识资料、课堂白板和对话区域同时纳入统一界面。

在当前版本中，系统通过预设的 8 步课堂脚本验证以下关键问题：教师如何主导课堂节奏，学生如何在多角色讨论中提出疑问，系统如何将重要内容沉淀到白板，资料如何通过来源按钮回溯到原始片段，演示组件和测验模块如何嵌入课堂主线之中。换言之，当前原型的主要任务是验证课堂交互框架，而不是证明后端平台已经完全部署。

## 2.2 技术路线与选型

在技术选型上，当前仓库以前端快速构建与高保真交互为优先目标，采用 `React 19 + TypeScript + Vite 6 + Tailwind CSS 4` 作为主体框架，通过 `motion` 处理卡片与消息动画，通过 `react-markdown + remark-math + rehype-katex` 完成数学公式与富文本渲染，通过 `lucide-react` 提供统一图标体系。这一组合使系统能够在较低工程成本下快速搭建具有教学展示质感的交互界面。

项目依赖中包含 `@google/genai` 与 `express`，说明系统在设计上为 AI 扩展能力和服务端扩展预留了技术接口；但从当前主流程实现来看，系统的核心演示并不依赖已部署后端服务，也不以真实 Gemini 调用作为运行前提。环境变量中仅明确了 `GEMINI_API_KEY` 与 `APP_URL`，其中 `GEMINI_API_KEY` 更适合作为后续扩展能力的可选配置，而非当前演示系统的必选运行条件。

## 2.3 总体架构与核心模块关系

系统总体架构可划分为四层。

第一层为**表现交互层**。该层直接面向教师与学生，包含三域工作台界面、卡片白板、来源抽屉、角色对话区域以及课堂控制按钮，负责课堂内容的可视化呈现与操作交互。

第二层为**课堂编排层**。该层负责课堂流程推进、消息队列播放、卡片生成、位置避碰、抽屉联动与状态重置，是当前仓库中最核心的流程组织层。`src/App.tsx` 即承担了该层的大部分职责。

第三层为**智能服务层**。从设计口径上看，该层可用于承载真实的多 Agent 协同、角色提示词策略、问题分发与生成式回答能力；但当前仓库并未把这一层完整落地为在线服务，现阶段更多体现为角色化对话表现与配置入口。

第四层为**知识数据层**。从设计上看，该层用于连接教师资料、课程题库、知识切片、引用索引与日志沉淀；但当前版本主要以静态前端数据和抽屉预览完成形态验证，尚未形成真实知识库入库与向量检索闭环。

表6 架构层次与实现边界表

| 架构层 | 当前仓库状态 | 后续扩展方向 |
| --- | --- | --- |
| 表现交互层 | 已实现 | 持续优化多端适配与课堂视觉语言 |
| 课堂编排层 | 已实现主要流程 | 接入更灵活的课堂脚本与教师控制面板 |
| 智能服务层 | 仅具角色化表现与配置入口 | 接入真实 Agent 编排与模型调用 |
| 知识数据层 | 仅有静态材料与预览形态 | 接入真实资料解析、索引与检索服务 |

## 2.4 核心业务闭环与运行机制

系统的课堂闭环可以概括为“教师供给资料 - 系统组织主题 - 多角色研讨 - 白板沉淀知识 - 仿真验证规律 - 测验完成反馈 - 小结与重置回看”。这一闭环体现了项目对智慧课堂的基本理解：课堂不是只输出答案，而是要把结论的形成过程、依据来源、演示验证与练习反馈一起组织起来。

在当前实现中，这一闭环由预设的 8 步课堂脚本驱动。用户点击底部候选按钮后，系统推进到下一课堂阶段；消息按照队列逐条播放，待本阶段消息全部完成后，再将对应的新卡片投放到白板，并为后续的来源点击、组件交互和测验作答提供入口。此种设计虽属于脚本化演示，但已经验证了“对话驱动白板、对白驱动资料联动、演示驱动练习反馈”的课堂式运行机制。

# 第三章   详细设计

## 3.1 界面设计

系统界面采用“三域协同课堂工作台”布局：左侧为来源抽屉，中部为知识白板，右侧为 Agent 对话区。该布局的设计原则不是单纯追求信息堆叠，而是通过空间分区对应课堂中的三类核心活动：资料查看、知识沉淀与讨论推进。左侧抽屉负责资料边界展示，中部白板负责板书与交互组件承载，右侧对话区负责课堂叙事推进与角色表达。

主界面顶部提供系统标题与“重置演示”入口，支持课堂回放。右侧对话区底部采用候选动作按钮推进下一课堂步骤，消息区支持逐条动画呈现与自动滚动。中部白板以点阵背景和大标题水印营造课堂展示氛围，所有卡片均以悬浮式模块存在，便于教师在讲解过程中自由拖拽与重组内容。

与传统聊天界面相比，本系统最重要的差异并不在消息气泡样式，而在于“消息不是终点，而是白板、引用、实验和测验的触发器”。每一轮课堂交流都可能同时影响多个界面区域，从而让课堂呈现出更加接近真实教学的工作台特征。

## 3.2 关键机制设计

### 3.2.1 教学资料锚定与来源溯源机制

系统在消息类型中定义了 `source` 字段，用于为消息绑定来源对象。当前来源对象包含 `id`、`title`、`page` 与可选高亮片段等信息。当教师或系统消息带有来源引用时，对话气泡下方会显示“引用”按钮；用户点击后，系统将 `activeSourceId` 传递给左侧抽屉，并根据正则匹配定位到对应材料类型。

`SourceDrawer` 组件中预置了 PPT 资料、题库资源和真题汇编等来源形态，并提供内容片段预览区，以便在课堂中形成“结论 - 引用 - 原文片段”的联动展示。该设计强化了课堂可解释性，也避免对话内容脱离教学资料独立悬浮。

需要说明的是，当前抽屉中的本地上传与链接导入逻辑采用 `setTimeout` 模拟，展示的是前端演示层面的资料挂载与预览效果，而非真实的教师资料解析、切片、索引与向量检索流程。因此，在完整系统设计口径中，`RAG` 更应被理解为未来的能力扩展方向；在当前仓库事实层面，更准确的表述应是“来源引用与知识边界可视化机制”。

### 3.2.2 多角色课堂协同机制

系统当前实现的角色枚举包括 `teacher`、`student_a`、`student_b`、`assistant` 与 `student`。其中，预设课堂脚本主要使用教师、学生 A、学生 B 与助教四类角色，用于分别承担讲授、提问、补充和总结等课堂职责。消息播放采用队列机制，系统在播放期间显示“正在输出”提示，并在一轮消息完成后统一投放对应卡片，从而维持课堂节奏的稳定性。

右侧对话区支持头像展示、名称映射、来源引用、候选动作推进和角色配置浮层。点击角色头像后，用户可以修改展示名称、查看系统提示词占位文本并切换“知识库引用偏好”，以此验证未来角色定制化与课堂策略配置的交互入口。

同时，系统也明确保留了当前边界：底部输入框处于禁用状态，发送按钮不可用，这意味着当前版本并不支持自由提问式实时对话。课堂推进主要由预设脚本和候选按钮控制，从而保证原型阶段的演示稳定性与流程可复现性。

### 3.2.3 知识白板与卡片编排机制

白板是本系统最核心的界面资产。当前卡片类型包括 `knowledge`、`simulator`、`quiz` 和 `summary` 四大类，其中“公式卡”和“易错点卡”并不是独立枚举，而是 `knowledge` 卡片在 `formula` 与 `pitfall` 字段上的不同呈现形态。这样的设计使卡片体系保持简洁，同时又能支撑课堂中的多种知识表达需求。

在交互层面，卡片支持拖拽、40 像素网格吸附、缩放、固定、折叠与编辑。知识卡片支持修改标题、正文、公式与易错点；总结卡片以更强视觉样式呈现课堂收束内容；测验卡片负责分题推进与答案反馈；仿真卡片则用于承载可操作实验组件。

为了避免白板上内容相互遮挡，系统在 `App.tsx` 中实现了卡片避碰与空位搜索机制。新增卡片会优先寻找下一个可用位置；拖拽或缩放后若发生重叠，系统会基于最小平移向量与启发式代价函数将相关卡片推开，并优先保持活动卡片或固定卡片的位置稳定。该机制使白板不仅仅是静态容器，而具备了一定程度的“课堂板书自动整理”特征。

### 3.2.4 交互仿真与测验一体化机制

系统将交互仿真与测验模块视为课堂主线的一部分，而不是外部跳转工具。当前预设课堂重点接入的是碰撞模拟器，支持完全非弹性碰撞与完全弹性碰撞两种模式。用户可以通过滑块调整两木块的质量与初速度，系统会实时计算总动量、碰后共速或分离速度，并以动画方式呈现碰撞过程，从而将动量守恒从静态公式转化为可观察现象。

除碰撞模拟器外，项目还在组件层实现了“冲量 - 动量变化模拟器”。该组件支持调整力、作用时间与质量，并实时计算 `I = F \cdot t` 与速度变化量。虽然当前 8 步主流程尚未默认投放该组件，但其实现已经为后续课堂脚本扩展预留了直接入口。

在测验部分，系统内置 5 道围绕动量定理与守恒应用的选择题，支持逐题切换、答案提交、正误反馈与解析展示。该机制与白板沉淀、小结卡片共同构成“讲解 - 演示 - 练习 - 回看”的课堂闭环，使课堂原型不止停留在展示层，而具备基本的教学反馈能力。

### 3.2.5 角色配置、扩展能力与安全边界

当前版本提供了角色配置浮层，允许用户修改角色展示名、查看角色人设指令占位和选择知识引用偏好。这一设计用于验证未来教师自定义课堂角色、调整回答策略和控制引用边界的可行交互形态。

但从工程边界看，当前系统仍是零持久化、无账号体系、无权限隔离、无日志治理的前端演示原型。角色配置并不会写入后端，来源抽屉并不具备真实知识库构建能力，课堂脚本也不是在线智能体即时协商生成的结果。因此，在描述完整系统能力时，必须把真实多 Agent 协同、资料入库、权限控制和课堂数据闭环明确标注为后续扩展方向，而不能混同为当前版本现状。

### 5.1.1 环境要求

当前仓库可证实的运行环境要求较为简洁，主要包括 `Node.js >= 18`、`npm` 与现代浏览器。项目主体为基于 `React 19 + TypeScript + Vite 6 + Tailwind CSS 4` 构建的前端演示系统，不依赖必须上线的后端服务即可完成课堂演示。

### 5.1.2 安装与启动

```bash
npm install

# 可选：如需预留 AI 扩展配置，可复制环境变量模板
cp .env.example .env.local

npm run dev
```

启动后访问 `http://localhost:3000` 即可进入系统主界面。当前环境变量中的 `GEMINI_API_KEY` 为可选项，用于后续扩展 AI 能力时配置，不是当前演示流程的必要前提。

项目提供如下基础命令：

```bash
npm run lint
npm run build
npm run preview
```

在使用过程中，用户可以完成以下操作：查看右侧多角色课堂对话；点击带引用的消息打开左侧来源抽屉；在中部白板拖拽、缩放、固定和折叠卡片；调整碰撞模拟器参数并播放动画；在测验卡片中逐题作答并查看解析；点击“重置演示”重新回到课堂初始状态。

当前版本中，底部输入框与发送按钮为禁用状态，这意味着系统重点服务于“课堂演示与流程验证”，而不是开放式自由问答。

# 第六章   项目总结

## 6.1 项目实施过程

本项目的实施遵循“问题驱动、场景收敛、原型先行、逐步验证”的思路展开。

**1. 需求收敛阶段**

团队首先聚焦课堂教学中的真实问题，即通用 AI 缺少课堂边界、课堂讲解与实验演示割裂、知识沉淀不连续、反馈链路不完整。在此基础上，项目没有从“大而全平台”切入，而是先聚焦一个可以被完整组织起来的课堂原型。

**2. 场景选择阶段**

在学科场景上，团队主动选择高中物理“动量定理 / 动量守恒”作为首个演示主题。该知识点同时包含概念辨析、公式推导、过程动画、实验解释与练习反馈，适合作为课堂交互框架的综合验证样本。

**3. 工作台搭建阶段**

在界面层面，项目构建了左侧来源抽屉、中部知识白板、右侧 Agent 对话区三域协同结构，并完成了消息播放、卡片生成、避碰重排、来源联动和角色配置浮层等核心交互。

**4. 课堂脚本编排阶段**

项目围绕动量主题设计了 8 步课堂流程，依次覆盖导入、辨析、推导、演示、守恒引入、碰撞验证、小结与测验。通过这一脚本化路径，系统得以验证“对话推动课堂”的基本可行性。

**5. 模块集成与演示验证阶段**

在完成白板、来源、仿真与测验模块后，团队将各部分整合为可本地运行、可重复回放的前端演示原型，并通过本地运行、课堂回放与交互验证持续修正措辞边界与模块衔接关系。

## 6.2 项目成果与创新点

本项目当前最重要的成果，是形成了一套可运行、可构建、可本地演示的智慧课堂交互前端原型。该原型并非传统意义上的静态作品展示页，而是已经具备课堂推进、知识沉淀、引用联动、仿真演示与测验反馈能力的交互系统。

| 动态知识白板 | 让知识沉淀与课堂推进同步发生 |
| 来源锚定机制 | 让资料依据在界面中可见、可点、可回溯 |
| 仿真与测验一体化 | 将实验与练习嵌入课堂，而非置于课后 |

## 6.3 项目价值

从教学价值看，本项目验证了智慧课堂系统不必从复杂平台建设起步，也可以先通过高保真前端原型验证课堂组织方式是否成立。对于教师而言，系统展示了资料、讲解、板书、演示和测验整合到同一工作台的可能性；对于学生而言，系统展示了多视角讨论、动态实验与即时反馈相结合的课堂体验。

从工程价值看，项目证明了在前端层面先行搭建课堂工作台，有助于提前识别教学流程中的关键交互点，减少后续后端建设中的抽象偏差。换言之，项目不仅在做一个展示作品，也在为后续工程化扩展验证交互边界和系统接口形态。

从竞赛与展示价值看，项目能够较完整地呈现“人工智能 + 教育”在课堂场景中的具体落点。它不是抽象概念性叙事，而是把课堂中的资料控制、讨论组织、知识沉淀和实验演示转化为可见、可操作、可回放的系统界面，因此具备较强的答辩展示力与阶段性说服力。

## 6.4 经验总结与未来展望

本项目的一个重要经验是：在智慧课堂场景中，课堂流程组织比单纯接入模型更重要。若没有清晰的课堂结构，再强的生成能力也很难转化为稳定、可信的教学体验。第二个经验是，界面与交互编排本身就是教学设计的一部分。白板如何沉淀知识、来源如何被看见、实验何时插入、测验如何衔接，这些问题本身就是课堂设计问题，而不是纯粹的界面美化问题。

第三个经验是，以前端原型先行是一种有效的系统验证路径。相较于一开始就投入大规模平台建设，先验证课堂工作台的表达能力、流程连贯性与用户理解成本，更有助于明确后续工程化重点。

未来，项目可以沿以下方向继续推进：

1. 接入真实教师资料解析、切片与检索服务，形成可用的 RAG 课堂知识层。
2. 引入真实多 Agent 编排与课堂控制策略，使角色协作从脚本化演示升级为动态课堂生成。
3. 扩展到更多学科场景，如化学实验、生物概念辨析与数学定理推导。
4. 建设教师后台、课堂记录与数据治理能力，支持长期使用与教学复盘。
5. 补充账号体系、权限控制与日志审计，使系统具备进一步平台化的工程基础。

总体而言，`Aurora Classroom · 智慧课堂交互演示系统` 已经完成了从课堂痛点识别到高保真前端原型验证的关键一步。它当前代表的是一个可信的阶段性成果：一个能够清晰表达智慧课堂交互形态、具备可运行演示能力，并为后续完整系统建设提供稳定边界的课堂交互原型。
